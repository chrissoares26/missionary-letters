### Breaking Changes / Important Findings

- **All Edge Functions called from `supabase.functions.invoke` must use `--no-verify-jwt`** ‚Äî the Supabase gateway JWT verification rejects valid tokens in this project. Auth is handled internally by each function via `auth.getUser()`.
- **`campaign_recipients` has no `owner_id` column** ‚Äî RLS ownership is enforced via `campaigns` join in policy. Do not add `owner_id` to inserts.
- **`campaign_send` idempotency guard blocks both `sent` and `sending`** ‚Äî a campaign in `sending` state (e.g. from a failed run) will need manual DB update to `approved` before retry. The outer catch now resets to `failed` as best-effort.

### Tips for Future Developers

- **Deploy all invoke-called functions with `--no-verify-jwt`** ‚Äî this is a project-wide pattern, not function-specific
- **`campaign_recipients` RLS** ‚Äî ownership is via `campaigns` join; no `owner_id` column exists on the table
- **Google refresh tokens** ‚Äî only sent on first `prompt=consent` grant; the OAuth callback conditionally includes it in the upsert payload to avoid `NOT NULL` failures on reconnect
- **Realtime subscription** ‚Äî always use `supabase.removeChannel(channel)` not `channel.unsubscribe()` to fully deregister
- **CSS tokens** ‚Äî must use `--action-primary`, `--border-default`, `--bg-canvas`, `--bg-muted`, `--text-primary`, `--text-secondary` ‚Äî not generic names like `--primary`, `--border`, etc.
- **`campaign_send` idempotency** ‚Äî checks `sent || sending` before `approved`; a stuck `sending` campaign requires manual DB reset to `approved` or relies on the outer-catch `failed` reset
